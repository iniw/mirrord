# AGENTS.md

Context for AI agents working in `mirrord/config` (`mirrord-config` and `mirrord-config-derive`).

## Scope

This file covers:
- `mirrord/config/src` (runtime config model, file/env resolution, validation, schema contract)
- `mirrord/config/derive/src` (the `#[derive(MirrordConfig)]` proc-macro and `#[config(...)]` flags)
- `mirrord/config/configuration.md` (published config reference generated from source docs)
- `medschool` crate workflow used to generate `configuration.md`

## Quick Reference

```bash
# Main config crate
cargo check -p mirrord-config --keep-going
cargo test -p mirrord-config

# Proc-macro crate used by mirrord-config
cargo check -p mirrord-config-derive --keep-going

# Schema maintenance (updates ../../mirrord-schema.json when stale)
cargo test -p mirrord-config check_schema_file_exists_and_is_valid_or_create_it -- --ignored --nocapture

# Regenerate configuration reference docs from source comments
cargo run -p medschool -- --input ./mirrord/config/src --output ./mirrord/config/configuration.md
```

## Architecture

### Two-layer model: file config vs generated config

- Most public config structs derive `MirrordConfig`.
- The derive macro generates a `*FileConfig` type (usually all fields become `Option<_>`) and an implementation that resolves final values into the runtime struct.
- `#[config(nested)]` means the field resolves via `FromMirrordConfig::Generator`.
- `#[config(toggleable)]` wraps nested generators with `ToggleableConfig`, allowing `true/false` shorthand.

### Source composition and precedence

- Value resolution uses `MirrordConfigSource` combinators (`.or`, `.layer`), mainly chaining env + file values.
- `ConfigContext` is the canonical environment access surface during generation/verification; it supports overrides and strict-env mode for tests.
- `LayerConfig::resolve` loads from `MIRRORD_CONFIG_FILE` if present, otherwise defaults.
- `LayerFileConfig::from_path` does key pre-resolution for Tera rendering:
1. `MIRRORD_ENV_KEY` from context if set;
2. `key` extracted from raw config file;
3. auto-generated key with marker prefix.
- Resolved configs are passed to downstream processes via `LayerConfig::encode`/`decode` and `MIRRORD_RESOLVED_CONFIG`.

### Validation and warnings

- `LayerConfig::verify` is the cross-field contract for conflicts and warnings.
- It enforces invariants across feature interactions (targetless/copy-target/operator, incoming filter exclusivity, fs buffer limits, startup retry bounds, etc.).
- Sub-verifiers are delegated to feature modules (`dns`, `outgoing`, `split_queues`).
- `Deprecated` and `Unstable` source layers add warnings into `ConfigContext` when those fields are used.

### Target and feature specialization

- `Target` supports shorthand string syntax and structured forms; parsing/compatibility rules live in `src/target.rs`.
- Target restrictions are split between:
1. target-level helpers (`requires_operator`, `requires_copy`);
2. `LayerConfig::verify` feature-dependent checks.
- Feature modules (env/fs/network/copy-target/split-queues/db-branches) each own their parsing defaults and analytics surface.

### Schema contract

- Schema source of truth is `schemars::schema_for!(LayerFileConfig)` in `src/lib.rs` tests.
- Any config shape/doc change requires regenerating `../../mirrord-schema.json` via the ignored schema test.
- `configuration.md` is generated by `medschool` from `./mirrord/config/src` docs/comments.
- Keep `configuration.md` aligned with public config behavior and field docs by re-running:
  `cargo run -p medschool -- --input ./mirrord/config/src --output ./mirrord/config/configuration.md`

## Operational Invariants

- During config generation and verification, do not read env directly with `std::env`; use `ConfigContext` or source wrappers.
- Preserve env-over-file precedence patterns when extending fields (CLI/frontends feed values through env vars).
- Keep `LayerFileConfig::from_path` key-resolution behavior consistent with `EnvKeyFileConfig::generate_config` (marker prefix is required to distinguish generated vs provided keys).
- Keep `Target` handling exhaustive across parser, display, schema variants, and compatibility helpers.
- For toggleable features, maintain correct `disabled_config` semantics so explicit env overrides still work.
- If a field affects behavior at runtime, also consider `verify` coverage and analytics collection.
- After any schema-impacting change, run the schema update test and ensure `mirrord-schema.json` is up to date.
- After doc-comment or config-doc changes, regenerate `configuration.md` with `medschool`.

## Change Workflows

### Adding a new config field

1. Add the field to the runtime struct and annotate with `#[config(...)]` as needed.
2. For nested structs, wire `FromMirrordConfig` and (if needed) toggleable behavior.
3. Add/adjust `LayerConfig::verify` or module-level `verify` checks for cross-field constraints.
4. Update analytics (`CollectAnalytics`) if the field should be reported.
5. Add tests for env/file precedence and validation behavior.
6. Run checks/tests, then regenerate schema if shape/docs changed.
7. Sync docs via `medschool`:
   `cargo run -p medschool -- --input ./mirrord/config/src --output ./mirrord/config/configuration.md`

### Adding a new target type

1. Add the target struct in `src/target/*`.
2. Add variant to `Target`, parsing in `FromStr`, display/discriminant handling, and schema `one_of`.
3. Update `requires_operator` / `requires_copy` and `TargetType::compatible_with`.
4. Update `LayerConfig::verify` rules if new interactions are required.

### Extending `#[config(...)]` derive behavior

1. Add/adjust flag parsing in `derive/src/config/flag.rs`.
2. Update generated definitions/implementations in `derive/src/config/field.rs` and container codegen in `derive/src/config.rs`.
3. Keep unsupported-flag diagnostics strict and mode-aware (container vs field).
4. Run `cargo check -p mirrord-config-derive --keep-going` and `cargo check -p mirrord-config --keep-going`.
